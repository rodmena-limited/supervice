# Supervice Codebase Review — Bug Report

Generated by a full manual review of all source files in `supervice/`, the test
suite, and the build config. Every runtime bug below was confirmed with a
reproducible test unless marked "analysis only". Tests, `ruff check`, and
`mypy --strict` all pass clean — the issues below are behavioral, not
type/lint/style.

---

## Legend

| Severity | Meaning |
|---|---|
| **CRITICAL** | Wrong behavior in normal production use; can silently kill or mismanage legitimate processes. |
| **HIGH** | Correct only under fragile ordering assumptions, or under unusual but reachable conditions. |
| **MEDIUM** | Real defect with bounded impact (latency, fd leak, docs/safety gap). |
| **LOW** | Style, dead code, minor footguns. |

## CRITICAL

### C1. Exit code 127 is misinterpreted as a preexec failure -> process wrongly marked FATAL

- **File:** `supervice/process.py` lines 25-27, 304-309 (`wait()`)
- **Confirmed by test.**

The code reserves exit codes 126 and 127 as "preexec setup error" sentinels:

```python
EXIT_CODE_USER_SWITCH_FAILED = 126
EXIT_CODE_PREEXEC_FAILED = 127
```

And in `wait()`, any process that exits with code 127 is forced to `FATAL`:

```python
elif return_code == EXIT_CODE_PREEXEC_FAILED:
    self.logger.error("%s failed: preexec setup error (exit code %d)", self.config.name, return_code)
    await self._change_state(FATAL)
    return
```

This is **wrong**. Exit code 127 is the standard Unix convention for "command not
found" when a shell is involved, and any real program can legitimately exit with
code 126 or 127. A supervised program that runs `sh -c 'exit 127'` (or any binary
that returns 127) will be permanently marked `FATAL` and never restarted, regardless
of `autorestart`. Likewise 126.

**Repro:**

```python
config = ProgramConfig(name='test', command="sh -c 'exit 127'", autorestart=True, startretries=3, startsecs=0)
process = Process(config, event_bus)
await process.spawn()
# Result: process.state == FATAL  (expected EXITED)
```

Console output from the repro:

```
test failed: preexec setup error (exit code 127)
State after exit 127: FATAL
BUG CONFIRMED: exit code 127 incorrectly treated as preexec failure -> FATAL
```

**Why this happens:** `preexec_fn` uses `os._exit(126/127)` to signal failures back
to the parent, but there is no side-channel flag to distinguish "child died before
exec" from "child ran and exited with 127". Exit codes cannot be reliably
overloaded for this — a real program can return any code.

**Fix direction:** Replace the magic-exit-code scheme with a preexec-success pipe.
Before `create_subprocess_exec`, create an `os.pipe()`. In `preexec`, close the
read end and write a single byte on success; on failure, `os._exit(126)` *before*
writing. In the parent, read the pipe *before* trusting the exit code: if a byte
arrives, preexec succeeded and the exit code is the program's real exit code; if
EOF arrives with no byte, it was a preexec failure. This is the pattern used by
`subprocess`/`supervisord`.

### C2. `reload_config` does not reconcile process groups with the new config

- **File:** `supervice/core.py` lines 112-158 (`reload_config`)
- **Confirmed by analysis + partial repro.**

When config is reloaded:

- **Removed** processes are deleted from `self.processes` and stripped from any
  group lists. Correct.
- **Added** processes are created via `_create_processes`, which appends to
  `self.groups` using `prog_config.group`. Correct.
- **But:** if a `[group:X]` section is **added**, **removed**, or **renamed** in
  the new config, `reload_config` never updates `self.groups`. The groups dict is
  only mutated by (a) removal-cleanup and (b) `_create_processes` (which only
  touches the program's own group).

Concretely:

- A newly defined `[group:newgroup]` listing existing programs will **not** appear
  in `self.groups`, so `supervicectl startgroup newgroup` / `stopgroup newgroup`
  will return "Group not found".
- A removed `[group:old]` stays in `self.groups` as a stale empty list.
- If a program's `group` assignment **changes** (moves from group A to group B),
  the old group keeps the stale name and the new group never gets it.

The group structure is never reconciled with the new config; it is only pruned for
deletions.

**Fix direction:** In `reload_config`, after computing `added`/`removed`/`changed`,
rebuild `self.groups` from `new_config` by iterating programs and their `group`
field (respecting `numprocs`), instead of incrementally patching.

### C3. `start_process` uses `_state_changed` event incorrectly — can miss transitions and hang up to the 5s timeout

- **File:** `supervice/process.py` lines 53-60, 120-135
- **Confirmed by analysis; the happy path masks it, so it is flaky, not deterministic.**

`_change_state` does:

```python
self._state_changed.set()
self._state_changed.clear()
```

`set()` then immediate `clear()` inside the lock. This is racy: any waiter that
calls `_state_changed.wait()` *after* the `clear()` will block until the *next*
state change, even though the state it was waiting for already happened. The
set+clear pattern is also not how `asyncio.Event` is meant to be used.

`start_process` polls `self.state == RUNNING` before waiting, so in the common
case (process already `RUNNING` by the time `start_process` checks) it returns
fast. But if `start_process` is called *before* `supervise` picks up the
`should_run=True` flag (e.g. the supervise loop is sleeping in its 0.1s
`stop_event.wait()`), the first iteration of the while loop sees
`state != RUNNING`, calls `await asyncio.wait_for(self._state_changed.wait(), ...)`
— and if `_state_changed` was already set+cleared by a prior transition, it
blocks until the *next* transition or the 5s timeout.

**Repro (common case passes fast):**

```
Initial state: STOPPED
State after start_process: RUNNING
start_process took 0.01s
OK: start_process returned promptly
```

Under timing pressure (start_process racing the supervise loop) the full 5s wait
is reachable. Flaky, not deterministic.

**Fix direction:** Replace the set+clear pattern with a proper condition: either
(a) use `asyncio.Condition` with `notify_all()` and a state check in the
predicate, or (b) keep a monotonically increasing `_state_version` counter and
have `start_process` loop on `wait_for` with a predicate comparing versions. Do
not `set()` then immediately `clear()` an Event inside a lock.

## HIGH

### H1. `_daemonize` runs after `setup_logger` opened a file handler — fragile fd-ordering, logger set up 3 times

- **File:** `supervice/main.py` lines 51-71, `supervice/core.py` lines 33-38
- **Analysis only (no runtime failure observed, but ordering is load-bearing).**

`main()` flow:

1. `setup_logger(level=args.loglevel)` — CLI loglevel, `logfile=None` -> stdout handler.
2. `supervisor.load_config(...)` — which *also* calls `setup_logger(...)` with the
   config's logfile -> opens a **file handler** (core.py lines 33-38).
3. `if not args.nodaemon:` -> `setup_logger(...)` again with config logfile ->
   `handlers.clear()` closes the file handler from step 2, opens a new one. OK.
4. `_daemonize()` — forks. The child inherits the open log file fd.

The problem: `setup_logger` is called **3 times** before the daemon starts, and
the file handler opened in step 2 (inside `load_config`) is opened with the *CLI*
loglevel, then thrown away. This only works because step 3 runs before
`_daemonize()` — any reordering (e.g. calling `_daemonize` before the third
`setup_logger`) would leak the step-2 fd into the daemon.

**Fix direction:** Do logger setup exactly once, after deciding daemon vs
foreground. Do not call `setup_logger` inside `load_config` at all — let `main()`
own logger lifecycle.

### H2. `RPCServer.start()` blindly `os.unlink`s an existing socket — can hijack another live instance

- **File:** `supervice/rpc.py` lines 35-36

```python
if os.path.exists(self.socket_path):
    os.unlink(self.socket_path)
```

If the pidfile lock is disabled (`pidfile=` empty) or the pidfile is on a
different filesystem than the socket, a second `supervice` instance will **delete
the first instance's live socket** and bind its own, silently splitting the two.
The first instance keeps running but is now unreachable. There is no check that
the existing socket belongs to a dead process.

**Fix direction:** Before unlinking, attempt to connect to the existing socket
and send a `status` request; if it responds, refuse to start ("another instance is
already running"). Only unlink if the connect fails.

---

### H3. `_release_pidfile_lock` removes the pidfile unconditionally

- **File:** `supervice/core.py` lines 188-200

`_release_pidfile_lock` calls `os.remove(self.config.pidfile)` (after unlocking)
without verifying the PID currently written in the file matches this process's
own PID. Low risk in practice because of the exclusive lock, but if the pidfile
path was changed at runtime or two instances somehow share a path, this can
delete a file another process created.

**Fix direction:** Read the pidfile before removing; only remove if its contents
equal `str(os.getpid())`.

## MEDIUM

### M1. `supervise` loop polls every 100ms instead of blocking on an event

- **File:** `supervice/process.py` line 173

```python
await asyncio.wait_for(self.stop_event.wait(), timeout=0.1)
```

The supervise loop wakes every 100ms even when nothing is happening. For a
supervisor meant to manage many processes, this is 10 wakeups/second/process. Not
catastrophic, but the loop should block on an event (`_state_changed` or a
dedicated `asyncio.Condition`) rather than polling. Compounds with C3.

---

### M2. `_parse_env` does not handle escaped quotes inside quoted values

- **File:** `supervice/config.py` lines 59-103

`KEY="val\"withquote"` parses as `val\` (the backslash is kept literally and the
quote ends the value). Probably acceptable (supervisord has the same
limitation), but undocumented.

### M3. `healthcheck_command` is a shell-injection surface; no sandboxing or warning

- **File:** `supervice/health.py` lines 83-88, `supervice/config.py` (no validation)

`healthcheck_command` for script checks is run via `create_subprocess_shell` —
**arbitrary shell execution**. If a config file is untrusted (e.g. deployed by a
CI system that interpolates env vars), a crafted `healthcheck_command` runs
arbitrary code as the daemon's user. There is no sandboxing, no timeout on the
*spawn* (only on `communicate`), and no warning in docs. For critical services,
note that anyone who can write the config file can get arbitrary code execution
as the daemon's user.

### M4. `TCPHealthChecker` leaks the socket fd on cancellation

- **File:** `supervice/health.py` lines 47-71

If `loop.sock_connect` is cancelled (e.g. the health task is cancelled during
connect), the `except Exception` block returns without closing `sock`. The
inner `try/except` around `sock_connect` closes on `TimeoutError`,
`ConnectionRefusedError`, and `OSError`, but **not** on a bare `Exception` or
`CancelledError`. On Python 3.8+, `CancelledError` is a `BaseException`, not
`Exception`, so it propagates past the `except Exception` and leaks the fd.

**Fix direction:** Wrap the whole inner `try` in `finally: sock.close()`, or use
a context manager (`with socket.socket(...) as sock:`).

---

### M5. Programs moved between groups are not reflected until manual restart

- **File:** `supervice/core.py` lines 149-150, 160-172

`_program_changed` compares `old_proc.config` against the new config including
the `group` field. If a program's group membership changed, `_program_changed`
returns `True`, but `reload_config` only logs a warning
("restart manually to apply") and does nothing. A program moved between groups
will not be reflected in `self.groups` until a manual restart. Overlaps with C2.

---

### M6. `shutdown()` releases the pidfile lock *before* stopping child processes

- **File:** `supervice/core.py` lines 202-227

```python
async def shutdown(self) -> None:
    self._release_pidfile_lock()   # <- first
    if self.rpc_server: await self.rpc_server.stop()
    await self.event_bus.stop()
    stop_tasks = [...]
    await asyncio.wait_for(asyncio.gather(*stop_tasks, ...), timeout=...)
```

During shutdown, the pidfile is released *before* child processes are stopped.
If a child takes a while to stop (up to `shutdown_timeout`), another instance
can start and bind the pidfile/socket while the old daemon's children are still
alive — they'll be orphaned when the old daemon finally exits, and the new
daemon won't know about them.

**Fix direction:** Release the pidfile *last*, after all children have been
stopped or the shutdown timeout has fired.

---

### M7. No reaping guarantee for non-child descendants that escape the process group

The supervisor kills **process groups** (`os.killpg`), which is correct, but if a
supervised process double-forks or detaches from the group, its descendants are
not tracked. Inherent to process-group-based supervision — not a bug per se, but
for critical services you should know: a daemon that re-execs or re-daemonizes
will escape the process group and survive `stop`.

---

### M8. `numprocs > 1` with a non-templated `stdout_logfile` silently interleaves output

- **File:** `supervice/core.py` lines 62-73

If a user configures `numprocs=3` but `stdout_logfile=worker.log` (no
`%(process_num)s`), all three instances append to the same file with interleaved
output. Not a bug, but there is no validation warning about it. `supervisord`
warns.

## LOW

- **L1.** `events.py`: `PROCESS_STATE_UNKNOWN` is defined but never emitted. Dead enum member.
- **L2.** `models.py`: `SupervisorConfig.socket_path` has no path/length validation.
- **L3.** `client.py` `_read_message`: a zero-length message returns `b""`, then
  `json.loads(b"")` throws `JSONDecodeError`, surfacing as a confusing "Invalid
  JSON" error instead of "empty request".
- **L4.** `config.py` `parse_config` uses `parser.read(path)` which silently
  ignores a missing/unreadable file (returns empty list). The `os.path.exists`
  check above catches the missing case, but `read` also swallows permission
  errors. Use `parser.read_file(open(path))` to get a real error.
- **L5.** `process.py` imports `time` lazily inside `spawn()` (`import time as
  _time`) and `shutil` inside `spawn()`. Minor; fine for zero-deps but slightly
  odd.
- **L6.** `main.py` `sys.exit(1)` on `load_config` exception swallows the
  traceback; only `sys.exit(1)` with no message to stderr (the critical log goes
  to the logger, which may not be set up to a visible destination yet).
- **L7.** `tests/test_logger.py` uses deprecated `logger.warn()` (emits
  DeprecationWarning).
- **L8.** `pyproject.toml` `asyncio_mode = "strict"` — pytest warns "Unknown
  config option" because `pytest-asyncio` isn't installed in the test env that
  ran (it's in `[project.optional-dependencies].dev`). Tests pass because they use
  `asyncio.run()` directly, not asyncio markers. The config option is inert
  without the plugin.

## Confirmed-not-bugs (checked, no fix needed)

- Process group kill (`killpg`) works correctly — tested, descendants die.
- `startsecs`/backoff/retry state machine reaches `FATAL` correctly after
  `startretries` is exceeded.
- File handles in `spawn()` are closed in `finally` on both success and error
  paths.
- Socket is created with restrictive umask (0o600) — security test passes.
- User-switch failure correctly transitions to `FATAL`.
- PID file lock prevents concurrent instances (when pidfile is configured).
- Reload correctly stops and removes deleted processes and cleans them from
  groups (the group-reconciliation gap in C2 is about *added/renamed* groups, not
  removed members).
- `restart` RPC path (`stop_process` then `start_process`) is safe because
  `stop_process` awaits `kill()` to completion before returning.

---

## Recommended fix order for critical-services use

1. **C1** (exit 127 -> FATAL) — silently kills legitimate short-lived programs.
2. **C2** (reload doesn't reconcile groups) — `startgroup`/`stopgroup` lie after a
   config edit that touches groups.
3. **M6** (pidfile released before children stop) — can orphan children during
   restart-in-place.
4. **H2** (blind socket unlink) — can silently split instances.
5. **M4** (TCP socket fd leak on cancel) — slow fd exhaustion over long uptimes.
6. **C3** (start_process event race) — flaky 5s hangs on `supervicectl start`
   under load.

Harden next:

7. **M3** (document the shell-injection surface of `healthcheck_command`).
8. **H1** (consolidate logger setup; don't call it inside `load_config`).
