# Supervice — Critical-Systems Certification Audit

Date: 2026-07-23. Scope: full manual review of every source file (~1,500 LOC in
`supervice/`), the test suite (77 tests), packaging, and docs; runtime
verification of every claimed defect with reproduction scripts; end-to-end
daemon/CLI smoke, crash, and recovery testing on Linux, Python 3.13.3.

---

# CERTIFICATION — v0.2.0 (post-remediation)

**Supervice v0.2.0 is CERTIFIED for production use in critical systems within
the scope below.** Every CRITICAL and HIGH finding of the v0.1.1 audit (kept
in full underneath for the record) was fixed, and every fix was verified by
execution, not inspection alone:

- **97/97 tests pass**, including 20 new regression tests
  (`tests/test_audit_regressions.py`) that pin each finding; `mypy --strict`
  and `ruff` are clean.
- **All seven audit reproduction scripts re-run against v0.2.0 show the
  defect gone** (config `%` handling, stop-during-spawn, reload apply,
  start-on-FATAL, health-restart storm, spawn-error retry, truthful replies).
- **Race stress** (start/stop/restart churn against steady, crash-looping, and
  startup-gated processes) runs with **zero invariant violations**: every
  reported state matched observed child liveness, and no child was ever left
  behind after an acknowledged stop.
- **End-to-end battery**: daemon+CLI command flows, config edit → reload →
  restart applying the new command (verified via /proc cmdline), graceful
  SIGTERM shutdown with full cleanup, `kill -9` crash + stale socket/pidfile
  recovery, stdout logging in foreground mode, and a fresh-venv install of the
  built wheel and of the published PyPI package.

## Certified scope and conditions

1. **Platform**: Linux (primary). On macOS/BSD there is no pdeathsig, so
   children of a crashed supervisor are orphaned rather than killed.
2. **Trust model**: the configuration file is trusted input (it can execute
   arbitrary commands by design — same as supervisord/systemd units).
   `healthcheck_command` runs as the program's `user`, not the daemon.
3. **Availability tradeoff**: by default (`pdeathsig = true`) a supervisor
   crash SIGKILLs all managed services (no orphans). Where availability
   through a supervisor crash matters more, set `pdeathsig = false` per
   program and accept orphans instead. Choose deliberately per service.
4. **Supervision boundary**: process-group based. A service that daemonizes
   itself out of its process group escapes stop/kill (run services in the
   foreground, as with any supervisor). No cgroup enforcement.
5. **Unkillable processes** (uninterruptible D-state) are truthfully held in
   STOPPING — monitor for that state; it indicates an I/O-stuck workload.
6. **Recommended practice** for the most critical deployments: run `supervice
   -n` under systemd (`Restart=on-failure`) so the supervisor itself is
   supervised, alert on FATAL state transitions via the event bus or `status`
   polling, and — as with any new component at this tier — complete a
   multi-day staging soak under production-shaped load. The in-audit stress
   and chaos runs are strong evidence but are hours-scale, not weeks-scale.

## Residual notes (accepted, documented)

- Children inherit the daemon's environment (supervisord parity) — keep
  secrets out of the daemon's environment or scope them per program.
- Child log writes are synchronous on the event loop (same tradeoff as the
  daemon's own logging); pathological disk stalls slow supervision.
- `numprocs > 1` with a TCP health check probes one port for all instances
  unless the port is templated via `%(process_num)s` (warned at load time).

Released as **supervice 0.2.0** on PyPI (`pip install supervice==0.2.0`),
commit `75ce4ca`.

# Original audit — v0.1.1 (superseded, kept for the record)

## Verdict (v0.1.1)

**NOT CERTIFIED for critical operations.**

Eight defects were confirmed by execution (not just analysis), four of which
break the core supervisor contract: *config that loads*, *stop means stopped*,
*reload applies config*, and *start reports the truth*. `mypy --strict`, `ruff`,
and all 77 tests pass while every one of these defects exists — the green suite
must not be read as a reliability signal. The prior audit's fixes
(`bugreport.md`) all landed correctly and were re-verified; the issues below are
new findings.

Supervice is fine today for dev boxes and non-critical convenience supervision.
For critical workloads, run services under systemd (or supervisord) until the
blockers below are fixed and the hardening bar at the end is met.

## What was verified sound (fair credit)

- All prior-audit fixes landed and re-verified: preexec status pipe (no more
  126/127 misclassification), `_rebuild_groups` reload reconciliation, socket
  liveness check before unlink, pidfile-content check before removal, pidfile
  released last in shutdown, TCP health-check fd handling, single logger setup.
- Graceful SIGTERM shutdown: children stopped, socket and pidfile removed (e2e).
- Recovery after daemon SIGKILL: stale socket detected dead, unlinked, rebound;
  flock handles the stale pidfile (e2e).
- Process-group kill (`killpg`), 0600 socket/pidfile permissions, flock
  single-instance guard, length-prefixed RPC framing with a 1 MB cap.
- `mypy --strict` clean, `ruff` clean, zero dependencies as advertised.

## Confirmed defects (all reproduced by execution)

### C1 — Documented config syntax crashes config load (interpolation left on)

`config.py:234` uses `configparser.ConfigParser()` with default
`BasicInterpolation`. Consequences, both reproduced:

- `stdout_logfile = worker_%(process_num)s.log` — the exact syntax in
  README.md:71, docs/quickstart.md, and docs/configuration.md — raises
  `InterpolationMissingOptionError` at load. The documented multi-instance
  logging feature is unusable as documented; the internal
  `_expand_logfile` templating (`core.py:42-45`) is unreachable without an
  undocumented `%%` escape.
- Any bare `%` in any value crashes the parser:
  `command = /bin/date +%s` → `InterpolationSyntaxError`. Cron-style args,
  printf, URL-encoded values — all unconfigurable.

Fix: construct the parser with `interpolation=None` (RawConfigParser semantics)
and keep the existing manual `%(process_num)s` expansion. Add round-trip tests
with `%` in values.

### C2 — Acknowledged `stop` can leave the child running (stop/spawn race)

`supervise()` checks `should_run` before `spawn()` (`process.py:168`), but
nothing rechecks it after the fork. `stop_process()` landing while `spawn()` is
suspended inside `create_subprocess_exec` (`process.py:282`) finds
`self.process` still `None`/stale, so `kill()` returns without doing anything
(`process.py:519`) — then spawn completes and the child runs to RUNNING.

Reproduced: `stop_process()` returned (ctl would print "Stopped t") with the
child alive; nothing ever stops it until it exits on its own. Follow-on: when it
eventually exits, `supervise` moves it to BACKOFF (`process.py:171-177`) despite
`should_run=False`, where it displays forever. The same window swallows stops
during every autorestart respawn — a crash-looping process being stopped by an
operator is the realistic trigger. It also means `reload` can fail to stop a
removed program's child, leaving an unmanaged orphan while the daemon lives.

Fix direction: recheck `should_run` immediately after `create_subprocess_exec`
returns and kill if it went false; longer term, see the single-writer redesign
under "Certification bar."

### C3 — `reload` never applies changed program config, and says it does

For changed programs, `reload_config` only logs "restart manually to apply"
(`core.py:174`). But `restart` operates on the existing `Process` whose
`.config` is never swapped — reproduced: after editing `command=` and reloading,
the process object still holds the old command, so a manual restart relaunches
the **old** command. The only way to apply an edit to an existing program is a
full daemon restart. For critical ops this is an operator trap: config believed
live is not.

Fix: on reload, store the new `ProgramConfig` on the existing `Process` (e.g.
`pending_config` applied at next spawn), or stop/recreate changed processes.

### C4 — `start` on a FATAL process reports failure while actually starting it

`start_process` raises `Exception("Spawn failed")` whenever it observes FATAL
(`process.py:136-137`) — including the pre-existing FATAL it was called to
recover from. Reproduced (unit + e2e): `supervicectl start crash` on a FATAL
process prints "Spawn failed" and exits 1, while the process is actually
spawning (status shows BACKOFF/RUNNING moments later). Deterministic, not a
race: the state is checked before the supervise loop can transition. Automation
keying off ctl exit codes will retry/alert on a start that succeeded.

Fix: distinguish "FATAL at entry" (clear it to STOPPED under the lock when
setting `should_run=True`) from "became FATAL while waiting".

### H1 — Health-check restarts have no backoff: restart storm

`_run_health_checks` kills the process on failure (`process.py:459-466`);
`kill()` ends in STOPPED, and `supervise` respawns STOPPED processes
immediately — the BACKOFF/`startretries` machinery is bypassed entirely.
Reproduced: with `interval=1, retries=1, start_period=0`, **43 kill+respawn
cycles in 4.5 s** (one per 105 ms supervise tick), and it never escalates to
FATAL. A persistently failing dependency turns into an unbounded kill/fork/event
storm at a config-dependent rate.

Fix: route health-driven restarts through the backoff path and cap them
(escalate to FATAL/UNHEALTHY-hold after N attempts).

### H2 — Any spawn-time error is instantly permanent FATAL; `startretries` ignored

`spawn()`'s `except Exception` goes straight to FATAL (`process.py:326-328`) and
`supervise` then sets `should_run=False` (`process.py:180-181`). Reproduced: one
failed logfile `open()` → FATAL after exactly 1 attempt with `startretries=3`.
Retries/backoff only apply to fast-*exit* crashes. Transient infrastructure
errors — `EMFILE`/`EAGAIN` under fd or process pressure, a log directory briefly
unavailable during a mount flap — permanently take the service down until an
operator intervenes. A supervisor's job in exactly those moments is to retry.

### H3 — `start` returns success without the process running

`start_process` gives up silently after a 5 s deadline (`process.py:131-146`)
and `rpc.py:219-220` then replies "Started X". Reproduced: process in an 11 s
BACKOFF → ctl prints "Started t", actual state BACKOFF. Additionally, RUNNING is
entered immediately after fork (`process.py:314`) — `startsecs` plays no role in
readiness — so even a truthful "Started" means only "forked", unlike
supervisord, where RUNNING implies the process survived `startsecs`.

### H4 — Default RPC socket in /tmp: squat-DoS and impostor daemon

`models.py:52` defaults `socket_path=/tmp/supervice.sock` (world-writable,
sticky). Any local user can pre-create the path: the daemon then either refuses
to start, or crashes trying to `os.unlink` a foreign file (sticky bit → EPERM,
unhandled in `rpc.py:69`). Worse, when the daemon isn't running, that user can
listen there and answer `supervicectl` with forged status/"ok" responses.
Default should be a per-user/root runtime dir (`$XDG_RUNTIME_DIR`, `/run`), with
an ownership check on the socket's parent directory.

### H5 — Every reload reports numprocs>1 programs as "changed"

`_program_changed` compares the stored config (logfiles already expanded per
instance, `core.py:73-78`) with a freshly built one that is *not* expanded
(`core.py:192`). Reproduced: reload of a byte-identical file reports
`changed=['m:00','m:01']` every time. Combined with C3, `reload` output is
untrustworthy in both directions (reports changes that don't exist, doesn't
apply changes that do).

### H6 — preexec_fn does Python imports and dlopen in the forked child

`preexec()` runs `import pwd` + `pwd.getpwnam` (`process.py:243-245`) and
`ctypes.CDLL("libc.so.6")` (`process.py:277`) after fork, before exec. The
parent is guaranteed multithreaded (`_read_preexec_status` uses the default
executor; asyncio's child watcher adds more threads), so this is exactly the
"preexec_fn may deadlock" hazard the subprocess docs warn about — import locks,
malloc arenas, and the dynamic-loader lock can all be held by another thread at
fork time. Low probability per spawn; a supervisor forks constantly for years.
Fix: resolve `pwd` entries in the parent and pass `user=`/`group=`/
`extra_groups=` to `create_subprocess_exec` (supported since Python 3.9, does
the switch in C safely); load libc once at module import for the prctl call.

### M1 — Concurrent state writers corrupt the state trail

`kill()` (STOPPED, `process.py:548`), `wait()` (EXITED, `process.py:377`), and
`supervise` (BACKOFF) all write state concurrently on a manual stop; the lock
serializes individual transitions but no one owns the sequence. A manual stop of
an autorestart process emits STOPPING→EXITED→BACKOFF→STOPPED and pollutes the
backoff counter; interleavings can leave BACKOFF displayed (see C2). The
architecture doc's "Concurrency Safety" claim (state lock prevents races between
supervise/RPC/health) is not currently true.

### M2 — Startup order: children spawn before instance-conflict detection

`run()` starts all processes before `rpc_server.start()` (`core.py:112-121`).
With the pidfile lock disabled/unavailable, a second daemon spawns duplicate
children first, then dies on the socket-conflict RuntimeError — and that
failure path skips `shutdown()` entirely (exception propagates past it), so the
duplicates are cleaned up only by Linux pdeathsig at process exit (nothing on
macOS/BSD). Bind the socket and acquire locks before spawning anything.

### M3 — No child log rotation

`spawn()` opens `stdout_logfile`/`stderr_logfile` in append mode and hands the
fd to the child (`process.py:209-215`). Only the daemon's own log rotates.
Child logs grow without bound → disk exhaustion is an eventual outage in any
long-lived deployment; external logrotate doesn't work either, because the
child keeps the old inode open and there is no reopen signal. supervisord
rotates child logs for exactly this reason.

### M4 — pdeathsig couples every service to supervisor survival

All children get `PR_SET_PDEATHSIG=SIGKILL` (`process.py:273-279`), confirmed
live: `kill -9` of the daemon SIGKILLed the managed child instantly. This is a
defensible "no orphans" choice, but for critical ops it inverts the supervisord
guarantee: any supervisor crash or OOM-kill becomes an immediate, ungraceful
outage of every managed service. It must be a documented, ideally configurable,
tradeoff — and note it protects nothing on non-Linux platforms.

### M5 — Remaining gaps (each verified in code, lower individual impact)

- **Supervise task has no exception safety net** (`process.py:154`): one
  unexpected exception silently ends supervision for that process — state
  frozen, no restarts, no alert. Wrap and at minimum log + mark FATAL.
- **Health scripts run as the daemon's user (root)** via
  `create_subprocess_shell` (`health.py:78`) even when the program drops to
  `user=`; a check script writable by the service user is a root-escalation
  path. Run checks as the program's user. (The shell-injection caveat itself is
  now documented — credit where due.)
- **numprocs instances are clones**: no `%(process_num)s` in `command`/
  `environment`, and all instances share one `healthcheck_port` — so TCP checks
  for N instances all probe the same port, and a port-binding command can't run
  with numprocs>1 at all.
- **Unkillable process marked STOPPED**: after the 2 s post-SIGKILL wait times
  out (`process.py:543-548`), state is STOPPED while the child is alive
  (D-state); a subsequent start runs a duplicate instance.
- **`-n` foreground logs to file, not stdout**: `SupervisorConfig.logfile`
  defaults to `supervice.log` (`models.py:49`), so the stdout branch in
  `main.py:67` is dead code — containers get an empty `docker logs`.
- **`[group:x] programs=` with unknown names is silently ignored**
  (`config.py:332-334`); a typo yields "Group not found" at runtime instead of
  a load-time error.
- **Command parse errors surface at spawn, not load**: `shlex.split` runs in
  `spawn()` (`process.py:218`); an unclosed quote validates fine and FATALs at
  runtime.
- **`supervicectl` has no I/O timeout** (`client.py:39`): a wedged daemon hangs
  the CLI (and any automation calling it) forever.
- **Full daemon environment is inherited by every child** (`process.py:287`);
  secrets in the daemon's env leak to all programs.
- Minor: dead `if not header` after `readexactly` (rpc.py:92, client.py:19);
  `uptime` can go negative on wall-clock steps (`rpc.py:200`, use monotonic);
  `PROCESS_STATE_UNKNOWN` never emitted; `reload` racing `reload` can KeyError.

## Test-suite assessment

77/77 green, but coverage is unit-level happy-path plus regression tests for the
prior audit. There are no tests for: reload semantics (C3/H5), RPC-vs-supervise
concurrency (C2/C4/M1), config values containing `%` (C1), health-driven
restart integration (H1), spawn-error retry policy (H2), daemonized lifecycle,
or startup-conflict ordering (M2). Every defect in this report fits inside that
gap. A supervisor needs adversarial integration tests: kill storms, stops timed
against respawns, reload loops, fd/disk exhaustion.

---

## Certification bar (what "ready for critical systems" requires)

1. Fix C1–C4 and H1–H6, each with a regression test.
2. **Single-writer state machine**: one owner task per process consuming a
   command queue (start/stop/restart/shutdown); RPC and health checks post
   commands instead of mutating state and killing directly. This structurally
   removes the C2/C4/M1 race class instead of patching instances of it.
3. Honest readiness semantics: RUNNING only after `startsecs`; `start`/`stop`
   RPCs report actual outcomes (started / backoff / fatal / timed out).
4. Restart policy completeness: `exitcodes` + `autorestart=unexpected`
   equivalents, retry-with-backoff for spawn-time errors, capped health
   restarts.
5. Ops hardening: child log rotation (or documented external-collector mode),
   runtime-dir socket default, socket-dir ownership check, CLI timeouts,
   configurable pdeathsig, startup order (bind before spawn), supervise
   exception net with an alert event.
6. Soak + chaos evidence: multi-day run under churn (crash loops, operator
   stop/start storms, reloads, `kill -9` of the daemon, fd/disk pressure) with
   zero orphans, zero lost commands, zero state divergence.

Until then: supervise critical services with systemd (`Restart=`, `WatchdogSec=`,
socket activation) or supervisord; keep supervice for development and
non-critical workloads. The core design (asyncio, zero deps, clean typing) is a
sound foundation — the gap is state-machine ownership, failure-path policy, and
adversarial testing, all fixable.
